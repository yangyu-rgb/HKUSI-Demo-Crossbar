# CrossBorder AI Demo 演示脚本

目标时长：约 5 分钟。

## 开场——20 秒

“CrossBorder AI 是为跨境运营方设计的预测式调度平台：将口岸不确定性转化为班次、车辆、货运任务和协同决策建议。个人规划仍然保留，但企业运营控制塔是本次主演示。”

明确说明：页面时间为香港实时时间；口岸、天气和交通使用本地模拟数据，不代表真实口岸情况。

先直接打开 `/`，用英文主标题和双语说明在五秒内说明产品用途，并说明口岸态势无需登录即可查看。首页继续使用既有城市视频 Hero 和页面布局。点击“Open Operations Control Tower”后进入 `/login`，选择“Coach Dispatcher / 跨境巴士调度主管”并自动返回企业页；微信与 Google OAuth 只展示未来生产接口，不执行真实登录。

## 场景一：口岸态势——35 秒

1. 打开 `/`。
2. 指出逐字进入的主标题、双 CTA，以及 Hero 右侧四张玻璃指标：当前最畅通、压力最高、未来一小时上升最快口岸和有效反馈/异常数量。
3. 展示 Three.js 港深微缩城市沙盘：指出带 OpenStreetMap 署名的离线港深轮廓、主干道路、程序化建筑和四个口岸实际相对方位；悬停福田查看跟随状态与预计等待，点击后平滑聚焦并继续拖动镜头，再使用“返回总览”和自动巡航开关。说明颜色表示四档压力、双向粒子越密代表人流越大、越慢且在口岸两侧累积代表排队压力越高；演示方向键或鼠标操作后巡航暂停并于8秒空闲后恢复，同时明确动画不是测绘、导航或真实旅客轨迹。
4. 比较四张动态卡片中的排名、当前等待、压力条、一小时变化、三小时峰值、90% 区间和异常提示。
5. 在趋势图中切换聚焦口岸，说明背景拥堵阈值与所选口岸 90% 区间。
6. 使用热力矩阵横向比较四个时点，并指出当前最优、三小时后最优和最大变化风险。
7. 指出全局香港时钟、60 秒数据刷新、手动刷新以及模拟天气和交通提示。

核心信息：产品不仅显示现在是否拥堵，也展示拥堵趋势。

## 场景二：可解释路线决策——85 秒

1. 打开 `/planner`。
2. 先展示首次进入的无方案引导，再从可搜索的固定列表选择香港大学 → 深圳南山科技园，保留动态建议到达时间，预算 HK$100。
3. 使用“稳妥均衡”显式生成建议。
4. 展示推荐 SVG 路线、最晚出发、预计到达、缓冲、动态 90% 区间、风险和费用。
5. 点击任一口岸的“查看完整计算过程”，在弹窗中展示模型基础、场景、官方等级、众包、深圳核验和最终等待，再用右上角关闭。
6. 修改出发地、目的地、偏好或预算并重新生成。
7. 切换为深圳→香港，再生成一次，证明两侧地点和四口岸交通矩阵都能闭环。

核心信息：系统回答“何时出发”和“为什么这条路线更稳”，依据来自可审查的 Demo 统计计算，而不是黑箱结论。

## 场景二补充：未来场景实验室——50 秒

1. 打开 `/scenarios`，选择未来 14 天中的一天。
2. 点击“一键课堂演示”，自动设置暴雨、节假日和深圳湾高强度活动。
3. 点击“对比 AI 方案”，在同一页面展示默认/草稿推荐口岸和四口岸等待、风险差值。
4. 指出预览没有保存数据；再显式保存场景并点击“使用此场景规划”。
5. 在路线页展示 AI v2.2 基础值、透明场景系数、官方等级权重、众包校准、深圳核验和新的推荐口岸。

核心信息：AI v2.2 只用口岸、方向、时间和香港客流学习底数；天气、事件、官方等级和众包用透明公式逐步修正。

## 场景三：持久化众包闭环——40 秒

1. 打开 `/crowdsource`。
2. 输入等待时间、方向、通关类型和可选说明；表单不预填虚假输入。
3. 说明所有输入都是课堂 Demo 数据：单人15%、双人30%、至少三名高质量高共识反馈45%上限，不收集为现场真实训练标签。
4. 提交并展示动态积分、质量分、独立反馈者、共识等级、当前有效权重和有效截止时间。
5. 再次向同一口岸提交，展示 10 分钟重复提交拦截。
6. 返回 `/planner` 重新生成，展示 AI v2.2 原始等待、众包权重和校准后等待的变化。
7. 说明过期反馈仍保存在 SQLite，但不会继续显示或参与预测，直到执行 Demo 重置。

核心信息：用户现场反馈闭合预测校准循环。

## 场景四：订阅与本地通知——40 秒

1. 打开 `/alerts`。
2. 创建提醒并在已保存列表中查看下一次提醒时间。
3. 打开提醒预览，说明下一次有效通勤日、推荐口岸和最晚出发时间。
4. 切换出发前、异常拥堵和更优路线三类开关；修改路线或时间，也可以删除订阅。
5. 点击“运行本地告警周期”，在通知收件箱查看触发结果并标记已读。

核心信息：主动服务包含可管理、可持久化的订阅和通知闭环；演示明确说明这是本地适配器，不是外部邮件、短信或推送。

## 场景五：企业预测与调度方案——主演示

1. 以 Demo 操作员进入 `/business`；页面从空表开始，先指出 CSV、模板和手工新增入口，再点击“Load Demo Sample”载入 10 条经过校验的巴士任务。
2. 点击“Compare All 4 Scenarios”，用同一批任务横向比较普通工作日、节假日高峰、演唱会散场和台风恶劣天气的风险、车辆冲突、情景暴露与调整数量。
3. 选择台风场景并分析，指出“输入校验 → HGB 预测 → 场景校准 → 约束优化 → 方案完成”链路，以及逐口岸 90% 区间和模型/降级来源。
4. 在任务表查看原口岸、车辆、出发时间与动态建议；解释台风透明映射为雷暴天气、高事件影响和口岸容量限制，不是假装存在独立台风训练特征。
5. 选择并采用措施，展示本地通知草稿、CSV 执行清单与人工复盘；通知未真实发送，车辆也未被真实调度。
6. 切换物流调度，载入货运样例并比较四场景；指出深圳湾使用 HGB，莲塘/文锦渡显式降级。再切换口岸官方，确认只显示聚合压力并发布协调建议。
7. `/business/employees` 保留原员工接驳流程，但不作为主演示。

核心信息：产品不是只展示数据，而是提前发现风险、比较方案、形成执行草稿并留下可复盘记录。

## 场景六：模型原理与手机版——45 秒

1. 打开 `/model`，指出 V1 Demo 显示“可完整演示”。
2. 展示合成数据离线指标、四口岸 MAE 和运行时影子差异。
3. 展示 AI v2.2 技术公式和大白话解释，指出香港客流负责主特征、深圳快照只核验区间。
4. 打开 `/mobile`，先展示独立 `/mobile/login`，以个人通勤身份登录后进入移动系统；企业或运营身份不能直接进入个人移动功能。
5. 进入“规划”，在手机端生成四口岸建议，再点击“通关后反馈实际等待”提交关联反馈并返回重新计算。
6. 进入“场景”，用综合压力预设完成无副作用 A/B 推演；再进入“我的”，展示提醒、通知和模型简报。
7. 说明手机端与桌面端共用同一套本地 API、SQLite 和 AI v2.2，但拥有独立交互流程；等待分钟仍是课堂 Demo 估算。

核心信息：模型的价值是课堂中可运行、可变化、可解释；同一能力可以在独立手机流程中闭环，但不是生产准确率声明。

## 场景七：运营分析——30 秒

1. 从移动端退出登录，回到网页登录页选择“Demo 操作员”，再打开 `/operations`。
2. 展示刚才产生的预测运行、众包质量结构、错误信号、业务操作、模拟订阅数和 Demo MRR。
3. 切换24小时与7天窗口，指出 SQLite、Provider、身份和通知仍是课堂适配器。

核心信息：同一套预测能力不仅服务个人路线，也能为 B2B/B2G 运营人员提供可追踪的运行视角。

## 场景八：套餐与模拟结账——35 秒

1. 打开 `/pricing`，切换月付与年付，比较个人、团队和企业三档能力。
2. 选择 Professional，指出页面没有银行卡或真实支付字段。
3. 确认模拟结账，展示本地收据、订阅状态与续期日期。
4. 返回 `/operations`，展示模拟订阅数和 Demo MRR 已更新。

核心信息：商业化方向已从文字说明升级为可操作购买流程，但不伪装成真实收入或支付接入。

## 结束与重置——25 秒

“CrossBorder AI 的商业重点不是向个人用户收导航费，而是把跨境等待风险转化为巴士、物流和口岸管理方可执行、可复盘的调度协同。”

下一次演示前确认已登录“Demo 操作员”，使用页头重置按钮，按当时香港时间恢复反馈和订阅种子，并清空通知、企业运营方案、协调建议、员工接驳方案、影子观测和审计动态数据。

移动端页头提供“网页版”返回键，“我的”页面提供退出登录；生产构建支持 PWA manifest、安装入口与离线说明页，但预测 API 在离线状态下不会伪造结果。

应用只使用仓库内可复现的官方快照；演示不依赖现场网络或采集器。
