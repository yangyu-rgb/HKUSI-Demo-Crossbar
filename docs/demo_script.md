# CrossBorder AI Demo 演示脚本

目标时长：约 5 分钟。

## 开场——20 秒

“普通地图能规划交通，却看不到未来口岸排队的不确定性。CrossBorder AI 将口岸等待预测、众包校准和主动提醒放进同一个跨境决策流程。”

明确说明：页面时间为香港实时时间；口岸、天气和交通使用本地模拟数据，不代表真实口岸情况。

## 场景一：口岸态势——35 秒

1. 打开 `/`。
2. 比较当前等待卡片和四条折线组成的未来三小时趋势图。
3. 指出全局香港时钟、60 秒数据刷新、手动刷新以及模拟天气和交通提示。

核心信息：产品不仅显示现在是否拥堵，也展示拥堵趋势。

## 场景二：可解释路线决策——85 秒

1. 打开 `/planner`。
2. 选择香港大学 → 深圳南山科技园，保留动态建议到达时间，预算 HK$100。
3. 使用“稳妥均衡”生成建议。
4. 展示推荐 SVG 路线、最晚出发、预计到达、缓冲、动态 90% 区间、风险和费用。
5. 展开“查看预测依据”，展示时间匹配历史基线、样本数量和衰减后的众包权重。
6. 修改出发地、目的地、偏好或预算并重新生成。
7. 切换为深圳→香港，再生成一次，证明两侧地点和四口岸交通矩阵都能闭环。

核心信息：系统回答“何时出发”和“为什么这条路线更稳”，依据来自可审查的 Demo 统计计算，而不是黑箱结论。

## 场景二补充：未来场景实验室——50 秒

1. 打开 `/scenarios`，选择未来 14 天中的一天。
2. 将天气改为暴雨或雷暴，添加只影响某一口岸的展会/演唱会事件并设置高强度。
3. 保存后点击“使用此场景规划”，跳转路线页。
4. 展示 AI v2 主预测、场景版本、相对默认场景等待变化和新的推荐口岸。
5. 返回场景实验室恢复当天默认，再次规划并对比结果。

核心信息：AI v2 将天气、事件、方向和时间作为模型输入；场景变化会改变等待预测，再由路线优化层生成不同方案。

## 场景三：持久化众包闭环——40 秒

1. 打开 `/crowdsource`。
2. 输入等待时间、方向、通关类型和可选说明；表单不预填虚假输入。
3. 说明“真实现场”和“建模同意”默认不勾选，普通 Demo 反馈不会进入 V2 快照；需要演示真实标签闭环时再明确勾选。
4. 提交并展示动态积分、质量分、可信等级、数据类别和有效截止时间。
5. 再次向同一口岸提交，展示 10 分钟重复提交拦截。
6. 返回 `/planner` 重新生成，展示有效众包数量和预测依据中的平均质量分。
7. 说明过期反馈仍保存在 SQLite，但不会继续显示或参与预测，直到执行 Demo 重置。

核心信息：用户现场反馈闭合预测校准循环。

## 场景四：订阅与本地通知——40 秒

1. 打开 `/alerts`。
2. 创建提醒并在已保存列表中查看下一次提醒时间。
3. 打开提醒预览，说明下一次有效通勤日、推荐口岸和最晚出发时间。
4. 切换出发前、异常拥堵和更优路线三类开关；修改路线或时间，也可以删除订阅。
5. 点击“运行本地告警周期”，在通知收件箱查看触发结果并标记已读。

核心信息：主动服务包含可管理、可持久化的订阅和通知闭环；演示明确说明这是本地适配器，不是外部邮件、短信或推送。

## 场景五：可编辑企业方案——40 秒

1. 在页头切换为“企业管理员”，打开 `/business`。
2. 导入员工 CSV 并展示行级校验；编辑批次默认偏好/预算，并为一名员工设置个人覆盖后生成方案。
3. 展示汇总风险、员工级出发建议及实际采用的偏好和预算。
4. 使用风险/口岸筛选，导出当前结果或已保存方案 CSV。
5. 再次生成，然后从近期方案历史中载入一次输入。

核心信息：同一预测能力可以支撑可重复的企业调度流程。

## 场景六：V1 模型与边界——30 秒

1. 打开 `/model`，指出 V1 Demo 显示“可完整演示”。
2. 展示合成数据离线指标、四口岸 MAE 和运行时影子差异。
3. 指出 AI v2 控制课堂场景预测，异常时自动降级；AI v1 保留为影子对照。
4. 展示真实数据门槛仍未满足，说明课堂合成场景指标不能作为真实口岸效果声明。

核心信息：V1 功能完整与 V2 数据就绪是两套独立判断，不能用生成数据绕过真实验证。

## 结束与重置——25 秒

“CrossBorder AI 的商业重点不是向个人用户收导航费，而是把跨境等待风险转化为巴士、物流和企业 HR 可执行的调度服务。”

下一次演示前切回“Demo 操作员”，使用页头重置按钮，按当时香港时间恢复反馈和订阅种子并清空通知、企业历史、影子观测和审计动态数据。

## 备用材料

- 60–90 秒屏幕录制。
- 每个路由页面一张截图。
- 一张架构图：缓存 JSON + SQLite → Service → API → 查询 Hook。
- 本地服务失败时使用 `docs/openapi.json` 和静态种子数据。
