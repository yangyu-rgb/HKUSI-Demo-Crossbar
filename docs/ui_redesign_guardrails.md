# UI 大改功能与信息保护清单

本文件是后续全项目 UI 风格整改的硬性边界。视觉提示词可以改变页面“如何呈现”，但不得在未获得明确授权时改变页面“提供什么信息、执行什么行为、谁可以访问”。

## 允许直接调整的视觉层

- CSS 变量、CSS Modules、配色、字体、字号、间距、圆角、描边、阴影和背景。
- 不承载业务含义的装饰图形、SVG、Canvas、Three.js 场景和纹理。
- 保持语义与内容不变的布局包装、响应式断点和卡片排列。
- 不阻塞操作的进入、退出、悬停、聚焦和数据切换动效；必须支持 `prefers-reduced-motion`。
- 图标外观和装饰性文案排版；图标按钮必须保留可访问名称。

## 未经明确授权不得改变

- `frontend/src/App.tsx` 中的路由、登录保护和角色权限。
- API 路径、请求字段、响应字段、TanStack Query 缓存及失效逻辑。
- 表单字段、默认值、固定地点 ID、校验、提交、清除、编辑、删除和导出行为。
- 页面已有的指标、单位、置信区间、风险、数据来源、模型版本、校准过程和 Demo 声明。
- 登录回跳、退出、模拟结账、本地收据、反馈关联、提醒周期和移动端返回网页版行为。
- 标题层级、表单标签、`role`、`aria-*`、键盘操作、焦点恢复和 44px 最小交互区域。
- `X-Demo-Persona-ID`、本地会话/套餐/安装提示的存储键及所有业务状态。
- `lack.md` 与项目文档已经确定的课堂 Demo 边界；禁止将本地模拟描述为真实认证、支付、收入或口岸实测。

如风格模板要求删减信息、合并表单、改变导航、替换图表指标或重写业务文案，必须先单独列出差异并获得用户确认，不能把它混入“仅改 UI”的提交。

## 每轮 UI 改造流程

1. 先记录基线：`git status --short --branch`、相关页面截图、`npm test`、`npm run test:e2e` 和 `npm run build`。
2. 将用户风格提示词拆成设计令牌、页面骨架、组件外观、动效和响应式五层；先列出任何可能触碰信息合同的部分。
3. 优先修改令牌、共享布局和 CSS Modules；仅在布局需要时调整 JSX 包装，不移动查询、Mutation 或业务状态。
4. 每次只改一个页面族：公开桌面、登录/商业、个人业务、企业/运营、移动端；完成一组就做回归。
5. 对照本文件逐页检查标题、字段、指标、按钮、权限、免责声明和键盘路径。
6. 运行 Vitest、Playwright + axe、生产构建和 `git diff --check`；涉及 API 行为时额外运行后端测试与契约检查。
7. 在 `project_memory.md` 记录本次视觉范围、明确未改变的功能、验证和剩余页面。

## 风格提示词交接模板

用户后续只需提供风格参考，Agent 应补齐而不是猜测业务变化：

```text
目标风格：
参考链接/截图/模板：
期望关键词（如：高科技、玻璃拟态、编辑感、深色、强动效）：
优先改造页面：
必须保留的品牌元素：
不喜欢的现有视觉：
动效强度（低/中/高）：
是否同时改移动端：

硬性要求：严格遵守 docs/ui_redesign_guardrails.md，只改显示效果；任何信息删减、字段变化、功能变化必须先征得确认。
```

如果参考模板的组件数量少于本项目信息量，应扩展模板的视觉语言来容纳现有内容，而不是删掉内容以迁就模板。

## 验收门槛

- 所有现有业务路由和角色守卫保持不变。
- 关键用户闭环的按钮和字段数量不减少，页面指标及单位不丢失。
- 桌面 1440/1024/768px 与移动 375px 无横向溢出。
- 纯键盘可完成登录、规划、弹窗关闭、结账、提醒和移动退出。
- `prefers-reduced-motion` 下仍可读、可操作；动效停止后信息不缺失。
- axe WCAG A/AA 无严重问题，前后景文字对比达到 WCAG AA。
- 自动化回归、生产构建和文档记录全部通过后，才算完成一轮 UI 改造。
